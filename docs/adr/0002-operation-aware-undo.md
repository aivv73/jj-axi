# Operation-aware undo semantics

jj-axi defines bare `undo` as reversing the latest user-visible repository mutation, not blindly restoring the immediate operation parent. It skips synchronization-only and prior undo operations, preserves newer synchronized working-copy content, and fails on divergent operation history because there is no unique latest mutation. This deliberately adds operation classification above `jj op restore` so agents can safely ask one deterministic question: “reverse my last VCS action.”

Explicit `undo --to <op-id>` restores exactly the selected historical state and treats a sole current operation as an idempotent no-op. With one operation head, the target must be that head or its ancestor. With divergent heads, explicit restore is the resolution mechanism: the target may be any operation reachable from at least one current head, including a head itself, and the restore resolves the repository to one new head. Unknown, ambiguous, wholly unrelated, and active-workspace-invalidating targets fail without mutation. Repository and workspace initialization are foundation operations rather than undo candidates; bare undo stops before removing them. Responses always expose all source operation IDs. Repeated bare undo continues backward; redo is not implicit.

Undo is strictly local. It never reverses pushes or other external effects and reports known unreverted effects. Repository mutations commit Jujutsu state before colocated-Git synchronization and working-copy update. If either post-commit step fails, jj-axi returns a structured partial result naming the failed step rather than attempting another automatic operation. Unknown operations created by other tools are conservatively classified as user-visible mutations; only known synchronization-only operations are skipped.
